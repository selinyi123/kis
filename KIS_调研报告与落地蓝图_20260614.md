## 0. 执行摘要（先读这段）

**一句话判断**：你不需要"从零造一个知识系统"。你手里已经有三块工业级资产——
1. **DPMS-Platform** 的 *Provider/Adapter 插件化 + 发现管道 + 价值/风险评分 + 事件日志事实源*；
2. **prompt-performance-engine（PPE）** 的 *契约/Schema 纪律 + 确定性编译器与模型判断分离 + 证据天花板*；
3. **ClipVault** 的 *双端采集 + 事件日志幂等同步 + Secret Guard + Obsidian/GitHub 入库*。

把这三块"换皮重组"，再用 Star 里 **Agent-Reach（连接器）+ MediaCrawler（国内平台）+ crawl4ai（网页→Markdown）+ MemOS（记忆/检索）+ codegraph（代码图谱服务 Codex）** 填补缺口，KIS 的 v0.1→v1.0 几乎是"组装"而非"发明"。

**核心结论**：KIS = **ClipVault 的采集底座 + DPMS 的管道/评分内核 + PPE 的证据纪律 + MemOS 的记忆运行时**。第一阶段（4–6 周）即可交付"多源采集→统一知识卡片→分类标签→项目关联→Obsidian 同步"的可用闭环。

# 第一部分　
## 1.1 DPMS-Platform 深度拆解（KIS 的内核来源）

**主流程**（README 原文）：
```
发现活动 → 解析目标 → 评估价值/风险 → 选择账号 → 排程执行
→ probe / dry-run / shadow-run → 安全门禁 → real-run
→ 证据保存 → 通知 → 复盘 → 策略建议
```
把这条线做"知识化映射"，几乎就是 KIS 的主流程（见 3.3）：
```
发现内容 → 解析正文 → 评估价值/质量 → 去重归类 → 排程处理
→ 清洗/摘要/打标 → 价值门禁 → 入库
→ 证据留存 → 通知 → 主题聚类 → 项目关联建议
```

**模块清单与可复用判定**：

| DPMS 模块（`core/app/...`） | 干什么 | KIS 复用价值 |
|---|---|---|
| `providers/`（bilibili/douyin/weibo/xiaohongshu + external_api + importer） | **发现层适配器**：发现→反钓鱼校验→ValueScorer→入库管道 | ⭐⭐⭐⭐⭐ 直接演化为 KIS 的**连接器层** |
| `services/discovery.py` | 扫描 tracked_sources → 关键词命中 → 校验 → 规范化 URL → 打分 → **幂等去重入库** | ⭐⭐⭐⭐⭐ KIS 的**采集调度+去重**骨架 |
| `knowledge/scoring.py` | 纯函数评分：`build_data_maturity`（数据成熟度 0-100/4 档）、`build_learning_gaps`（P0/P1/P2 缺口）、`confidence_score`、`account_reputation` | ⭐⭐⭐⭐⭐ 改造成 KIS 的**价值评分+知识缺口检测** |
| `knowledge/service.py` | 多数据源聚合成 summary（窗口化、data_sources 声明、profiles） | ⭐⭐⭐⭐ KIS 的**知识仪表盘聚合**模板 |
| `risk/intelligence.py` | `tighten_action` 安全原语：只能让决策**更保守不能更宽松**；forecast 只收紧不放松 | ⭐⭐⭐⭐ 改造成 KIS 的**自动操作门禁**（如自动归档/删除前的保守闸门） |
| `event_store/service.py` | V4 事件日志：账号/任务/通知/风险/Worker/探针/登录事件 | ⭐⭐⭐⭐⭐ KIS 的**事实源**（与 ClipVault outbox 同构） |
| `experiment/engine.py` | A/B 实验引擎 | ⭐⭐⭐ 用于评分/摘要策略调参 |
| `governance/policy.py` | 危险操作策略门禁 | ⭐⭐⭐ KIS 删除/外发的合规闸门 |
| `learning/feature_store.py` + `model.py` | 特征存储 + 轻量学习模型 | ⭐⭐⭐ v2 个性化价值评分 |
| `strategy/engine.py` + `transition/engine.py` | 策略建议 + 状态机迁移 | ⭐⭐⭐⭐ KIS 知识卡片**生命周期状态机** |
| `services/scheduler.py` + `recovery_daemon.py` | 排程 + 恢复守护 | ⭐⭐⭐⭐ 长期运行的采集排程 |
| `api/knowledge.py` `api/risk_intel.py` ... | FastAPI 路由 | ⭐⭐⭐ KIS 后端 API 模板 |
| `dashboard/dist`（React 前端） | 中英/明暗主题、事件时间线 | ⭐⭐⭐ KIS Web 控制台直接借壳 |

**四个"资产级"模式（这是 bil/dpms 给 KIS 留下的真正遗产）：**

1. **Provider/Adapter 插件化**：发现层（`providers/`）与执行层（`adapters/`）解耦，每个平台一个适配器，遵循统一管道契约。→ KIS 的"插件化连接器"直接就是这个范式。
2. **确定性管道 + 评分门禁**：discovery 把"扫描→校验→规范化→打分→去重入库"做成纯函数可测（`test_knowledge_scoring.py` 等 12 个测试无需 DB）。→ KIS 的清洗/去重/评分必须同样**纯函数可测**。
3. **事件日志事实源（Event Store V4）**：所有状态变化是事件，可复盘、可重算。→ KIS 的知识资产变更同样事件溯源。
4. **tighten-only 安全原语**：自动决策只能更保守。→ KIS 的"自动删除/自动外发/自动覆盖"必须套这个闸门（防止 AI 误删知识）。

## 1.2 prompt-performance-engine（PPE）— 证据纪律资产

PPE 表面是"Prompt 优化器"，但它的**工程纪律**正是 KIS 处理"AI 生成内容（摘要/标签/评分）"时最缺的东西：

| PPE 设计原则 | 对 KIS 的直接借鉴 |
|---|---|
| **契约层（Contract Layer）**：版本化 request/artifact/evaluation/evidence schema，默认拒绝未知字段 | KIS 的**统一知识卡片**必须是版本化 schema，迁移函数随 schema 走 |
| **确定性编译器 vs 模型判断分离**："编译器确定性，模型只产候选" | KIS 的**清洗/规范化确定性**，**摘要/标签由 LLM 产候选**，二者不可混 |
| **静态审计引擎**：20 个对抗注入用例（A01–A20，注入/越权/零宽字符/base64 隐藏指令/XML 突破…） | KIS 采集**外部内容前必须过注入审计**（网页/推文可能藏 prompt 注入，污染你的 LLM 摘要） |
| **证据天花板 E0/E1 + claim 三级**（optimized/verified/top_tier） | KIS 的**价值评分要标注证据等级**：纯启发式 vs LLM 判断 vs 用户确认，不能把 AI 猜测当事实 |
| **schema 驱动、profile 数据化**（`domain_profiles.json`） | KIS 的**分类标签体系用数据文件**驱动，不写死在代码 |

> 🔑 关键迁移：PPE 的 `adversarial_cases/` 整套对抗用例，**直接搬到 KIS 当"入库前内容安全闸门"的测试集**——这是把"采集开放互联网"和"喂给 LLM"安全连接的护城河，外面的方案普遍没有。

## 1.3 clipvault-personal（ClipVault）— 采集底座资产

ClipVault 已经是一个**完成度 100%（12 切片全绿、128 pytest、双端发版 v1.0.0）**的采集系统，它就是 KIS 的"剪切板/本地文档/双端采集"那一路源，且已解决最难的工程问题：

| ClipVault 已有能力 | KIS 中的角色 |
|---|---|
| 双端剪切板同步 + Android IME/Share Target 采集 | **来源连接器**：剪切板 + 移动端剪藏 |
| 事件日志式同步（outbox seq + content_hash 双幂等） | **去重 + 同步内核**（与 DPMS Event Store 同构，KIS 统一用这套） |
| Secret Guard 三道闸门（密钥不进 FTS/不外发） | **采集安全**：KIS 抓外部内容更需要它 |
| 分类/Suggestion(SUG-1.1)/Context Action | **分类标签 + 推荐**的确定性 v1 |
| Obsidian 入库（frontmatter+wikilink）+ GitHub JSONL 私备 | **KIS 的同步归档层**直接复用 |
| 桌面零运行时依赖（纯 stdlib + ctypes，Web UI 用 http.server） | KIS 桌面主节点同款轻量底座 |
| 角色纪律：Architect 写切片规格、Builder 实现、HANDOFF.md 事实源 | **KIS 沿用同一套交付纪律** |

> 🔑 关键决策：**KIS 不要新建采集底座，而是把 ClipVault 升级为 KIS 的"采集 + 同步 + 入库"内核**，新增的只是"更多来源连接器 + 知识加工管道"。这能省掉至少 60% 的工程量，并复用已验证的幂等/安全/同步机制。

## 2.2 按系统分层的"主选 + 替代方案"决策表

> 决策原则：**能复用你自己的就不引外部；外部优先选"零 API 费 + 本地优先 + 活跃"的**。

### 节点 ① 多源连接器（Collection）
- **主选**：`Agent-Reach`（Twitter/X、Reddit、YouTube、GitHub、B站、小红书、RSS、网页一把梭，零 API 费）。它本身就是"插件化连接器"理念，与 DPMS `providers/` 同构 → 作为 KIS 外部源的统一入口。
- **替代/补充**：
  - 国内深度采集（评论、创作者主页、登录态）→ `MediaCrawler`（小红书/知乎/B站更全）。
  - GitHub Stars → 直接 `gh api users/<you>/starred`（你已验证可用，无需第三方）。
  - Twitter/X 书签 → Agent-Reach；若失效，导出 X data → 本地解析。
  - 剪切板/本地文档/移动剪藏 → **ClipVault**（你已有）。
  - 网页剪藏 → crawl4ai（见节点②）。
- **自建兜底**：`BaseConnector` 契约（DPMS providers 抽象），任何源失效都能 1 天内补一个适配器。
- **风险**：Agent-Reach 依赖第三方接入方式会换代（作者承诺维护）；MediaCrawler 明确"仅学习用途、禁商用"，**KIS 是个人自用合规边界内**，且只抓你自己已收藏的公开内容。

### 节点 ② 网页/正文清洗（Clean & Extract）
- **主选**：`crawl4ai`（网页→干净 Markdown，LLM 友好，开源本地）。
- **替代**：Agent-Reach 自带网页读取（轻量场景）；`trafilatura`/`readability`（纯 Python 兜底，无浏览器）。
- **稳健性增强**：`CloakBrowser` 替换 Playwright 内核应对反爬。
- **复用你自己的**：ClipVault 的 Secret Guard 在此清洗阶段同步运行（脱敏）。

### 节点 ③ 去重（Dedup）
- **主选（自建复用）**：ClipVault `content_hash` + DPMS discovery 的 `canonicalize_url` + `insert_if_new`。URL 规范化 + 内容哈希双重幂等。
- **增强**：近似去重用 `simhash`/`minhash`（轻量）或 embedding 余弦相似（需向量库，见节点⑦）。
- **不引外部**：去重是确定性逻辑，自建最可控。

### 节点 ④ 摘要 / 分类 / 标签（Enrich，LLM）
- **主选**：本地 Claude/Codex 调用 + **PPE 的证据纪律**（摘要=候选、标注证据等级、过注入审计）。
- **加工素材**：`book-distill`（长文蒸馏）、`stop-slop`（去 AI 味）、`ljg-skills`/`dbskill`/`pm-skills`（分类/诊断 skill）。
- **替代**：`last30days-skill` 的合成范式用于"多条同主题卡片→一篇综述"。
- **关键约束**：分类标签体系用 **数据文件驱动**（PPE `domain_profiles.json` 范式），不写死。

### 节点 ⑤ 价值评分（Score）
- **主选（自建复用）**：DPMS `knowledge/scoring.py` 改造——0-100 分 + 4 档（cold/warming/usable/ready），多信号加权（来源权威度、与在做项目相关度、新鲜度、被引用次数、用户行为）。
- **v2 升级**：DPMS `learning/feature_store` + `model` 做个性化（你点开/复用过的卡片反哺评分）。
- **证据等级**：纯启发式 / LLM 判断 / 用户确认 三级标注（PPE claim 范式）。

### 节点 ⑥ 主题聚类（Cluster）
- **主选**：embedding + HDBSCAN/KMeans（`sentence-transformers` 本地嵌入）。
- **替代/增强**：交给 MemOS 的图结构（节点⑦）天然形成主题簇；或 `Understand-Anything` 思路做"卡片→知识图谱"聚类可视化。

### 节点 ⑦ 存储 / 记忆运行时（Memory & Retrieval）
- **主选**：`MemOS`——图结构记忆（可检视可编辑，非黑盒 embedding）、统一 store/retrieve/manage API、混合检索、有 local-plugin（本地优先）。**这是 KIS"可被 AI 调用"的核心运行时**。
- **替代（轻量起步）**：SQLite + FTS5（你 ClipVault 已用）+ `sqlite-vec`/`chroma` 向量。**v0.1 先用 SQLite+FTS，v0.3 接 MemOS**，避免一上来过重。
- **决策**：MemOS 是"目标态"，但**第一阶段不要被它阻塞**——先用你已验证的 SQLite+FTS 跑通闭环，再平滑迁移（保留 schema 兼容）。

### 节点 ⑧ 项目关联（Link to Projects）
- **主选（自建）**：知识卡片打 `project` 字段 + 反向索引；规则：标签/关键词/embedding 相似命中你的活跃项目（DPMS、ClipVault、KIS 自身、prompt-engine）。
- **DPMS 借鉴**：`strategy/engine.py` 的"建议"模式——KIS 给出"这条知识可能服务于 X 项目的 Y 模块"。

### 节点 ⑨ 更新跟踪（Track Updates）
- **主选（自建复用）**：DPMS `scheduler.py` + `discovery.should_scan`（按 interval 重扫）+ ClipVault 事件日志（diff 检测）。GitHub repo 监 `pushedAt`/release；网页监 content_hash 变化。
- **替代**：RSS（Agent-Reach 自带）做被动更新通知。

### 节点 ⑩ 同步归档（Sync & Archive）
- **主选（自建复用）**：ClipVault 的 Obsidian（frontmatter+wikilink）+ GitHub JSONL 私备，**100% 直接复用**。
- **不引外部**：你这块已是成品。

### 节点 ⑪ 服务 Codex / AI 调用（Serve）
- **主选**：`codegraph`（本地代码知识图谱，自动同步，原生支持 Claude Code/Codex/Cursor）→ 让 KIS 的"代码类知识"直接可被 Codex 检索。
- **配合**：MemOS local-plugin 暴露记忆给 Agent；KIS 自身暴露 MCP/CLI（`kis search`/`kis recall`）。
- **替代**：`Understand-Anything`（偏可视化探索）；`anysearch-skill`（实时联网补充）。

### 节点 ⑫ 自动研究报告（Research，v2+）

## 3.3 统一知识卡片模型（KnowledgeCard v0.1 — 立即可冻结的 schema）

```jsonc
{
  "schema_version": "0.1.0",
  "id": "kc_<ulid>",                  // 稳定主键
  "content_hash": "sha256:...",       // 去重幂等键（DPMS/ClipVault 范式）
  "source": {
    "connector": "github_stars|clipboard|web|twitter|zhihu|bilibili|xhs|local",
    "url": "https://...",
    "captured_at": "2026-06-14T...Z",
    "raw_ref": "outbox_seq 或原始事件ID"
  },
  "content": {
    "title": "",
    "body_md": "",                    // crawl4ai 清洗后的 Markdown
    "lang": "zh|en",
    "media": []                       // v2: 视频/图文解析产物
  },
  "enrichment": {
    "summary": "",                    // LLM 候选，非事实
    "tags": [],                       // 数据文件驱动的标签体系
    "category": "",                   // 一级分类
    "topics": [],                     // 聚类主题
    "value_score": 0,                 // 0-100
    "value_level": "cold|warming|usable|ready",
    "evidence_level": "heuristic|llm|user_confirmed"  // PPE 证据等级
  },
  "linkage": {
    "projects": ["DPMS-Platform","ClipVault","KIS"],  // 项目关联
    "related_cards": ["kc_..."],      // 卡片间关联
    "wikilinks": ["[[...]]"]          // Obsidian 双链
  },
  "lifecycle": {
    "state": "inbox|cleaned|enriched|linked|archived|stale",
    "created_at": "...", "updated_at": "...",
    "last_checked_at": "...",         // 更新跟踪
    "version": 1
  },
  "safety": {
    "secret_redacted": true,          // ClipVault Secret Guard
    "injection_scanned": true         // PPE 注入审计
  }
}
```
> **冻结建议**：v0.1 先定这 7 个顶层域，新增字段走 schema 迁移函数（PPE 范式），永不破坏已入库卡片。

## 3.4 知识生命周期（状态机，复用 DPMS transition/engine）

```
inbox ──clean──▶ cleaned ──enrich──▶ enriched ──link──▶ linked ──▶ archived
   │                                                                    ▲
   └──(注入/秘钥命中 → quarantine 隔离)         (更新检测变化 → stale)──┘
```
每次迁移写事件日志，可复盘、可重算。删除/外发/覆盖前过 `tighten_action` 保守门禁。

## 3.7 验收标准（v0.1 阶段整体 Definition of Done）

1. **三源贯通**：GitHub Stars + 网页剪藏 + 剪切板，三类源都能产出统一 KnowledgeCard。
2. **去重幂等**：同一内容重复采集不产生重复卡片（content_hash 验证）。
3. **可搜索**：SQLite FTS5 全文检索 + 标签/分类/项目过滤可用。
4. **可同步**：卡片同步到 Obsidian（双链）+ GitHub JSONL 私备。
5. **schema 稳定**：KnowledgeCard v0.1 冻结，校验拒绝未知字段，有迁移函数。
6. **纯函数可测**：清洗/去重/打标/评分核心逻辑无 DB 依赖单测（DPMS 范式），CI 全绿。
7. **HANDOFF 事实源**：每切片完成更新 HANDOFF + push + Obsidian（ClipVault 纪律）。
